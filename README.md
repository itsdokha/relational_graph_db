# Домашняя работа: реляционная и графовая (сетевая) модели данных

**Предметная область — «Кинобаза»:** фильмы, люди (актёры и режиссёры) и жанры.
Одна и та же модель реализована двумя способами и сравнивается:

- **Реляционный подход** — PostgreSQL (язык SQL);
- **Сетевой (графовый) подход** — Neo4j, модель LPG (язык Cypher).

> Графовая БД выбрана **Neo4j** — это самый наглядный вариант: шаблон запроса
> `(:Person)-[:ACTED_IN]->(:Movie)` визуально повторяет сам граф, у неё лучшая
> документация и встроенный браузер с визуализацией. Именно Neo4j используется
> и в примере преподавателя.

---

## 1. Предметная область

Рассматриваем факты участия людей в фильмах. Человек может быть **актёром**
(сыграл роль в фильме) и/или **режиссёром** (снял фильм). Каждый фильм отнесён
к одному или нескольким **жанрам**.

### Сущности

| Сущность | Описание | Атрибуты |
|----------|----------|----------|
| **Person** (человек) | актёр и/или режиссёр | `person_id`, `full_name`, `birth_year`, `country` |
| **Movie** (фильм) | полнометражный фильм | `movie_id`, `title`, `release_year`, `rating`, `duration_min` |
| **Genre** (жанр) | жанр фильма | `genre_id`, `name` |

Связи между сущностями:

- **ACTED_IN** — человек сыграл роль в фильме (свойство `character` — имя персонажа);
- **DIRECTED** — человек срежиссировал фильм;
- **HAS_GENRE** — фильм относится к жанру.

### Правила в предметной области

1. Человек может сниматься в любом количестве фильмов.
2. В фильме может сниматься множество актёров.
3. Один и тот же человек может быть и актёром, и режиссёром (напр. Тарантино).
4. У фильма может быть один или несколько режиссёров.
5. Фильм относится как минимум к одному жанру, жанров может быть несколько.
6. Один человек не может играть **одну и ту же** роль в одном фильме дважды
   (но может сыграть несколько разных ролей).

### Упрощения в модели

1. Учитываем только две функции человека — актёр и режиссёр (нет сценаристов,
   операторов, продюсеров и т.д.).
2. У фильма один год выпуска (без учёта разных дат премьеры по странам).
3. Рейтинг — одно усреднённое число (без разбивки по источникам/годам).
4. Рассматриваем только полнометражные фильмы (без сериалов и сезонов).
5. Имя персонажа (`character`) необязательно — может отсутствовать.

### Вопросы к предметной области (≥5)

Именно на них отвечают запросы в обеих БД (см. §5):

1. **В каких фильмах снимался актёр X** (Leonardo DiCaprio)?
2. **С кем актёр X чаще всего снимался вместе** (со-актёры)?
3. **Какой жанр самый популярный** (по числу фильмов)?
4. **Что порекомендовать** зрителю, которому понравился фильм X (по общим жанрам)?
5. **Какова дистанция (число Бэйкона)** — кратчайшая цепочка совместных съёмок
   между двумя актёрами?
6. **Кто самый плодовитый режиссёр** (по числу снятых фильмов)?
7. **Контроль качества данных** — есть ли фильмы без жанра или задвоенные роли?

Вопросы 2, 4 и особенно 5 — про *связи между связями*; на них удобно отвечать
именно графовой моделью (см. выводы в §6).

## 3. Реляционная модель (PostgreSQL)

### ER-диаграмма в DBML

Полный листинг — `models/schema.dbml` (вставляется на dbdiagram.io для картинки).
Ключевая идея: связи **многие-ко-многим нормализованы** через ассоциативные
таблицы, поэтому в схеме остаются только связи **1:M**:

| Связь предметной области | Тип | Ассоциативная таблица |
|--------------------------|-----|-----------------------|
| Person — Movie (актёр) | M:N | `role` (ACTED_IN) |
| Person — Movie (режиссёр) | M:N | `direction` (DIRECTED) |
| Movie — Genre | M:N | `movie_genre` (HAS_GENRE) |

Чтобы не появлялись «лишние правила» (как в примере преподавателя про Заказ без
Пациента), внешние ключи в ассоциативных таблицах объявлены `NOT NULL` —
строки `role`/`direction`/`movie_genre` не могут существовать без своих сущностей.

### Реализация и наполнение

- `relational/init/01_schema.sql` — DDL (таблицы, ключи, индексы);
- `relational/init/02_data.sql` — данные (14 человек, 10 фильмов, 11 жанров);
- `relational/queries.sql` — запросы-ответы на все 7 вопросов.

## 4. Графовая модель (Neo4j, LPG)

### Labeled Property Graph

```
(:Person {person_id, full_name, birth_year, country})
(:Movie  {movie_id, title, release_year, rating, duration_min})
(:Genre  {genre_id, name})

(:Person)-[:ACTED_IN {character}]->(:Movie)
(:Person)-[:DIRECTED]->(:Movie)
(:Movie)-[:HAS_GENRE]->(:Genre)
```

Картинка LPG — `diagrams/lpg.png`. В отличие от реляционной модели, **связи
M:N не требуют ассоциативных таблиц** — отношение в графе является «гражданином
первого класса» и хранится прямым ребром (а атрибут роли `character` живёт
прямо на ребре `ACTED_IN`).

### Реализация и наполнение

- `graph/01_constraints.cypher` — ограничения уникальности (= PRIMARY KEY + INDEX);
- `graph/02_data.cypher` — те же данные, что в реляционной БД;
- `graph/03_queries.cypher` — запросы-ответы на все 7 вопросов.

## 5. Запуск

```bash
docker compose up        # или docker-compose up   (для Compose v1)
```

Поднимаются два контейнера и наполняются автоматически:

| БД | Адрес | Доступ |
|----|-------|--------|
| PostgreSQL | `localhost:5432` | db=`movies` user=`movies` pass=`movies` |
| Neo4j Browser | <http://localhost:7474> | user=`neo4j` pass=`moviesmovies` |

**Запросы к реляционной БД:**

```bash
docker exec -i movies_pg psql -U movies -d movies < relational/queries.sql
```

**Запросы к графовой БД** — открыть Neo4j Browser и выполнять запросы из
`graph/03_queries.cypher` по одному (визуализация графа прилагается), либо:

```bash
docker exec -i movies_neo4j cypher-shell -u neo4j -p moviesmovies < graph/03_queries.cypher
```

Полезный запрос для скриншота всего графа в браузере:
`MATCH (n)-[r]->(m) RETURN n,r,m`

## 6. Сравнение подходов и выводы

| Критерий | Реляционная (SQL) | Графовая (Cypher) |
|----------|-------------------|-------------------|
| Связи M:N | через ассоциативные таблицы (`role`, `direction`, `movie_genre`) | прямым ребром, доп. таблиц не нужно |
| Атрибут связи (`character`) | отдельное поле в таблице `role` | свойство прямо на ребре `ACTED_IN` |
| «Соседи соседа» (вопрос 2 — со-актёры) | `JOIN` таблицы `role` саму с собой | шаблон `(p)-->(:Movie)<--(co)` — читается как картинка |
| Обход переменной глубины (вопрос 5 — число Бэйкона) | **рекурсивный CTE** `WITH RECURSIVE` + ручной BFS, массив пути, защита от циклов | один вызов `shortestPath(...-[:ACTED_IN*..10]-...)` |
| Читаемость запросов на связях | падает с ростом числа `JOIN` | шаблон визуально совпадает со структурой данных |
| Целостность / ограничения | сильная: FK, `NOT NULL`, `UNIQUE`, типы | гибкая схема, ограничения на ключевые поля |
| Агрегации и табличные отчёты (вопросы 3, 6) | очень удобно и привычно | тоже удобно (`count`, `ORDER BY`) |

### Конкретные выводы

1. **Чем «связнее» вопрос — тем выгоднее граф.** Вопросы 2 (со-актёры), 4
   (рекомендации) и 5 (кратчайший путь) в Cypher выражаются почти дословно
   формулировкой вопроса. В SQL вопрос 5 потребовал рекурсивного CTE с ручным
   обходом в ширину, массивом пройденного пути и защитой от циклов — сравните
   объём `relational/queries.sql` (вопрос 5) и одну строку `shortestPath` в Cypher.

2. **Реляционная модель сильнее в целостности и в табличной аналитике.** Внешние
   ключи и `NOT NULL` строго гарантируют, что не появится «роль без фильма» или
   «фильм без режиссёра как сущности». Для простых агрегатов (вопросы 3, 6) оба
   подхода равноценны.

3. **Стоимость связей платится в разное время.** В SQL связь M:N — это лишняя
   таблица и `JOIN` на каждом запросе (стоимость на чтении). В графе связь хранится
   физически как ребро, поэтому обход «по связям» не требует соединений
   (дешевле на чтении, но схема менее строгая).

4. **Итог.** Для предметной области, где главная ценность — именно *связи*
   (рекомендации, сети знакомств/съёмок, пути), графовая модель даёт более
   короткие и наглядные запросы. Для строгой учётной системы с акцентом на
   целостность и табличную отчётность реляционная модель остаётся предпочтительной.
   В нашей «Кинобазе» большинство интересных вопросов — про связи, поэтому
   графовый (сетевой) подход здесь выглядит естественнее.
